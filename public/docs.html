Abstract Messenger — Documentation

Anonymous, end-to-end encrypted messaging on AbstractChain.

Overview

Abstract Messenger enables private chats using end-to-end encryption. Messages are transmitted as on-chain transactions on AbstractChain, and mirrored to an indexed datastore to enable fast retrieval and cross-device availability.

Key Advantages over Web2

  • Anonymity: Chats are address-based; no email/phone/SSO.
  • Ownership: Messages are tied to your wallet; no centralized accounts.
  • Security: Content is encrypted end-to-end using modern crypto (AES‑GCM).
  • Portability: History can be reconstructed from chain, independent of any single server.
  • Transparency: Message metadata is verifiable; payloads remain unreadable to others.

How Encryption Works

For each conversation, a shared key is derived deterministically from both participants’ addresses. The payload is encrypted using AES‑GCM with a random IV. Only participants can compute the correct key and decrypt.

  • Key derivation: PBKDF2 + conversation salt (SHA‑256 of sorted addresses).
  • Cipher: AES‑GCM (256‑bit), random IV per message.
  • Prefix: enc:v1:<iv_base64>:<ciphertext_base64> for safe transport.

Realtime and Reliability

  • Realtime: Supabase Postgres changes subscriptions for instant updates.
  • Fallback: Periodic polling ensures delivery when realtime is unavailable.
  • Recovery: A light chain scanner can reconstruct missing messages from on-chain data.

Profiles and Identity

  • Avatars and usernames are pulled from Abstract Portal by wallet address.
  • If no on-chain username is set, a local nickname can be applied.
  • When no nickname exists, the UI falls back to a shortened address.

Presence and Read Status

  • Online status reflects when the other participant has your chat open.
  • Typing indicators appear in real time while the other user is composing.
  • Read receipts show when the latest outgoing message has been seen.

Session Sending

Session sending allows pre-authorized message transmissions without per-message wallet prompts. Implementation typically relies on smart wallet session keys or permissioned delegates. This UI includes a “Create Session” control and persistence, ready to be wired to wallet APIs that support sessions.

Interface

  • Liquid-glass visual system with layered blur and neon accents.
  • Mobile-friendly layout with stacked panels and touch-ready controls.

Privacy Notes

  • Payloads are unreadable without the conversation key.
  • Addresses and timestamps are public on-chain.
  • Consider using fresh addresses for stronger unlinkability.

Getting Started

  • Connect AGW.
  • Enter a peer address to start a conversation.
  • Write a message and sign the transaction.
